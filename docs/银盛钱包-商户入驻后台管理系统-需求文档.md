## 1. 产品概述

### 1.1 项目背景

银盛钱包入驻分为**个人账户**和**商户账户**两种类型：

#### 个人账户入驻流程（6步）
短信登录→填写资料→银盛审核→签约确认→绑定银行卡→设置密码

#### 商户账户入驻流程（5步）
登录→填写信息→银盛审核→签约→微信/支付宝认证

> **⚠️ 重要规则：账户类型重选机制**
> - **提交审核前（步骤1-2）**：用户可返回「账户类型选择」页重新选择个人或商户类型
> - **提交审核后（步骤3及之后）**：账户类型锁定，不可更换

用户在入驻过程中可能因各种原因中断申请。后台管理系统需要帮助运营人员：

1. **实时掌握申请进展**：清晰看到每个客户的入驻状态、已完成步骤、待完善的步骤
2. **主动触达客户**：对于中断或待完善的申请，可主动发送"继续入驻"链接，引导客户完成剩余步骤
3. **代为入驻操作**：对于无法自行完成操作的商户（如年纪大、操作困难），运营人员可直接在后台代为完成入驻

### 1.2 产品目标

| 目标 | 衡量指标 |
|------|----------|
| 提高入驻完成率 | 申请→完成 转化率提升 |
| 减少客群流失 | 中断客户重新激活率 |
| 提升运营效率 | 单客户跟进耗时 < 2分钟 |

## 3. 数据结构

### 3.1 入驻类型枚举（account_type）

| 类型值 | 含义 | 说明 |
|--------|------|------|
| `PERSONAL` | 个人账户 | 个人用户入驻，6步流程 |
| `MERCHANT` | 商户账户 | 商户用户入驻，5步流程 |

### 3.2 商户主表（merchant）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键 |
| account_type | enum | 入驻类型：personal（个人）/ merchant（商户） |
| phone | varchar(20) | 手机号（登录账号） |
| name | varchar(50) | 姓名（个人）/ 商户名称（商户） |
| id_card_no | varchar(20) | 身份证号（加密） |
| bank_card_no | varchar(30) | 银行卡号（加密）- 个人：储蓄卡 / 商户：对公账户 |
| bank_name | varchar(50) | 银行名称 |
| status | enum | 见 3.4 状态枚举 |
| current_step | int | 个人：1-6 / 商户：1-5 |
| apply_source | enum | 入驻来源：online（线上自申请）/ admin（后台代录入） |
| operator_id | bigint | 代操作运营ID（后台录入时记录） |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |
| **--- 商户特有字段 ---** | | |
| business_license_no | varchar(50) | 营业执照号（加密） |
| legal_person_name | varchar(50) | 法人姓名 |
| legal_person_id_card | varchar(20) | 法人身份证号（加密） |
| legal_person_id_front | varchar(255) | 法人身份证正面照URL |
| legal_person_id_back | varchar(255) | 法人身份证反面照URL |
| business_license_photo | varchar(255) | 营业执照照片URL |
| merchant_address | varchar(200) | 经营地址 |
| contact_name | varchar(50) | 联系人姓名 |
| contact_phone | varchar(20) | 联系人手机号 |
| settlement_bank | varchar(50) | 结算银行 |
| settlement_account | varchar(30) | 结算账号（加密） |
| wechat_auth_status | enum | 微信认证状态：pending/completed/failed |
| alipay_auth_status | enum | 支付宝认证状态：pending/completed/failed |
| merchant_type | enum | 商户类型：individual（个体工商户）/ enterprise（企业） |

### 3.3 步骤记录表（onboarding_step）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键 |
| merchant_id | bigint | 商户ID |
| step | int | 个人：1-6 / 商户：1-5 |
| step_status | enum | pending/completed/failed |
| step_data | JSON | 该步骤提交的完整数据 |
| completed_at | datetime | 完成时间 |

**步骤对照表：**

| 步骤 | 个人流程 | 商户流程 |
|------|---------|---------|
| 1 | 短信登录 | 登录 |
| 2 | 填写资料 | 填写信息（营业执照、法人、结算账户等） |
| 3 | 银盛审核 | 银盛审核 |
| 4 | 签约确认 | 签约 |
| 5 | 绑定银行卡 | 微信认证 |
| 6 | 设置密码 | 支付宝认证 |

### 3.3 状态枚举

| 状态值 | 含义 | 说明 |
|--------|------|------|
| `IN_PROGRESS` | 进行中 | 入驻流程未完成 |
| `PENDING_AUDIT` | 待审核 | 等待银盛审核 |
| `AUDIT_FAILED` | 审核失败 | 可重新提交 |
| `AUDIT_REJECTED` | 审核拒绝 | 需重新申请 |
| `COMPLETED` | 已完成 | 入驻成功 |

### 3.4 继续入驻邀请表（continue_invite）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键 |
| merchant_id | bigint | 商户ID |
| resume_from_step | int | 从第几步继续（1-6） |
| invite_token | varchar(64) | 邀请链接Token（唯一） |
| channel | enum | sms/wechat/copy |
| recipient | varchar(100) | 发送目标（手机号/微信openid） |
| sent_at | datetime | 发送时间 |
| expires_at | datetime | 过期时间 |
| opened_at | datetime | 打开时间（可空） |
| completed_at | datetime | 完成时间（可空） |

## 5. 代入驻操作说明

### 5.1 代入驻适用场景

| 场景 | 说明 | 操作方式 |
|------|------|----------|
| 老年商户不会操作 | 年纪大的商户，智能手机使用困难 | 运营电话指导 + 后台代操作 |
| 信息填写错误 | 商户填写的信息有误，需要修改 | 后台直接修正 |
| 客户在外地 | 商户不方便返回操作 | 后台代完成签约等步骤 |
| 批量入驻 | 多个商户统一入驻 | 批量导入 + 统一代操作 |

### 5.2 各步骤代操作说明

| 步骤 | 可代操作内容 | 所需材料 |
|------|-------------|----------|
| 1. 短信登录 | 发送验证码 → 运营输入完成 | 手机号 |
| 2. 入驻申请 | 代上传证件、代填信息 | 身份证照片、身份信息、银行卡 |
| 3. 银盛审核 | 仅审核，不可代操作 | - |
| 4. 签约确认 | 代上传签名图片 | 签名文件 |
| 5. 绑定银行卡 | 代填写银行卡信息 | 银行卡号、手机号 |
| 6. 安全设置 | 代设置支付密码 | 6位数字密码 |

### 5.3 代操作权限控制

```
角色：普通运营
  ├── 可代操作：步骤1、2、4、5、6
  └── 不可操作：步骤3（银盛审核）

角色：审核员
  ├── 可代操作：步骤1、2、4、5、6
  └── 可操作：步骤3（银盛审核）

角色：管理员
  └── 所有步骤均可代操作，包括撤回重做
```

### 5.4 代操作日志

每次代操作需记录：
- 操作人（运营人员ID）
- 操作时间
- 操作步骤
- 完成方式（自操作 / 代操作）
- 操作前后数据变化

## 6. 继续入驻链接机制

### 5.1 链接生成规则

```
域名：www.yinsheng.com/c/{token}

Token 生成规则：
- 32位随机字符串（UUID v4）
- 包含 merchant_id、resume_step、expires_at 加密签名
- 每次发送生成新 Token，旧 Token 失效
```

### 5.2 链接访问流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 客户点击 │ → │ 验证Token │ → │ 验证步骤 │ → │ 跳转页面 │
│ 邀请链接 │    │ 和有效期  │    │ 是否匹配  │    │ 对应步骤 │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                                      │
                        ┌─────────────┴─────────────┐
                        ↓                           ↓
                   [匹配]                      [不匹配]
                   跳转步骤                   弹窗提示：
                   继续操作                   "链接已过期，请重新获取"
```

### 5.3 邀请状态追踪

| 状态 | 说明 |
|------|------|
| `SENT` | 已发送，等待打开 |
| `OPENED` | 链接已打开 |
| `IN_PROGRESS` | 正在操作中 |
| `COMPLETED` | 已完成所有步骤 |
| `EXPIRED` | 链接已过期 |
| `CANCELLED` | 被新邀请覆盖 |

## 7. 非功能性需求

### 6.1 性能要求

| 指标 | 要求 |
|------|------|
| 列表页加载 | < 1秒（分页100条） |
| 详情页加载 | < 0.5秒 |
| 邀请链接生成 | < 0.2秒 |
| 支持并发 | ≥ 1000用户同时在线 |

### 6.2 安全要求

- [ ] 身份证号、银行卡号前端展示需脱敏（中间4位显示*）
- [ ] 身份证照片查看需二次验证（如：输入验证码）
- [ ] 审核操作需记录操作人、操作时间、操作IP
- [ ] 邀请链接 Token 不可猜测、不可伪造
- [ ] 敏感操作需日志审计

### 6.3 通知要求

- [ ] 审核结果通过：短信通知商户
- [ ] 审核结果拒绝：短信+站内信通知
- [ ] 签约超时提醒：第3天、第7天各提醒一次
- [ ] 入驻完成：短信通知恭喜
